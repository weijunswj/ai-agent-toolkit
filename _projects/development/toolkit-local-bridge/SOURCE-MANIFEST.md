## Repo Script Source

[repo/scripts/toolkit-local-bridge.cjs](../../../repo/scripts/toolkit-local-bridge.cjs) is the canonical shared bridge updater source. The native plugin hooks call that script for optional bridge autocheck, passive repo-local instruction managed-block preflight, repo auto-update, and enabled-target sync. It is not copied from private Codex or Claude plugin caches.

[repo/scripts/toolkit-staging-generations.cjs](../../../repo/scripts/toolkit-staging-generations.cjs) owns generation-scoped staging registration, identity validation, audit, current-operation cleanup, and exact approved reconciliation. It pre-registers immutable ownership before creating staging, never authorizes deletion from a matching name or dead PID alone, and leaves historical unmarked or ambiguous entries untouched.

Bridge version state is recorded independently for `repo`, `codex-plugin`, and `claude-plugin`. Same-source downgrade protection uses only the running source's recorded version; the monotonic `hub_version` remains reporting and older-cache compatibility state. Locked writes reread and merge the latest state so independently cached host copies cannot erase one another's entries. Existing cached copies are not repaired by source changes, so each affected native host must be refreshed after merge.

[repo/scripts/setup-codex-toolkit-plugin.cjs](../../../repo/scripts/setup-codex-toolkit-plugin.cjs) is the Codex-only native plugin install verifier and local marketplace wrapper runner. It validates `.agents/plugins/marketplace.json`, headless-safe `ON_USE` auth policy, `.codex-plugin/plugin.json`, installed Codex plugin state, the current Toolkit version, the installed-cache `SessionStart` hook, its cache-local exact Node runtime metadata, and package-critical cache freshness before setup reports success. The fingerprint includes the setup core and every direct setup dependency, including `toolkit-agent-control.cjs` and `claude-process-launch.cjs`; missing or stale same-version bytes require the supported refresh path. On Windows it atomically publishes and verifies runtime metadata before replacing the portable source command with the exact PowerShell launcher, so interruption leaves either the prior source hook or a valid runtime plus the prior hook. The installed command uses a direct PowerShell `-File` boundary so Codex Desktop's outer command host cannot strip nested script-text quoting; it does not depend on interactive-shell Node resolution or shell-style `${PLUGIN_ROOT}` expansion. Supported non-interactive inspection does not expose hook trust, so guidance routes the user to `/hooks` and identifies the installed plugin-cache copy as the executing hook source while the managed Git checkout remains the refresh source. It never reads or edits trust state manually and does not install or update Claude Code.

[repo/scripts/toolkit-codex-session-start.cjs](../../../repo/scripts/toolkit-codex-session-start.cjs) and [repo/scripts/toolkit-codex-session-start.ps1](../../../repo/scripts/toolkit-codex-session-start.ps1) form the Codex hook-safe launcher boundary. The Node launcher fixes the permitted hook arguments, derives the plugin root from its own package location, and converts thrown or returned non-zero optional-maintenance outcomes into one visible safe skip. The Windows launcher owns the fixed approved hook argument vector, reads only setup-generated cache-local runtime metadata, preserves standard input and bridge output, uses the exact verified Node executable, and exits zero after a visible safe skip when startup cannot proceed. Manual bridge commands retain their normal non-zero failures.

[repo/scripts/setup-claude-toolkit-plugin.cjs](../../../repo/scripts/setup-claude-toolkit-plugin.cjs) is the Claude Code-only native plugin install verifier and local marketplace wrapper runner. It validates `.claude-plugin/marketplace.json`, `.claude-plugin/plugin.json`, `.claude-plugin/hooks/hooks.json`, the exact installed `toolkit-local-bridge.cjs` SessionStart bytes and regular-file topology, and installed Claude Code plugin state via `claude plugin list --json` before setup reports installation freshness. Strict Agent enforcement is separate: it additionally requires host-reported trust and active hook execution, bound by activation-proof schema 3 to the exact plugin version, installed-cache identity, hook bytes, controller bytes, process-launch bytes, and Agent-hook bytes. Every bound path must be a non-symlink regular file. Setup cannot create that proof, schema-2 proof is stale and unsupported, and restart/trust instructions are never treated as completed. It uses supported plugin commands instead of editing Claude Code config by hand and does not install or update Codex.

[repo/scripts/setup-toolkit.cjs](../../../repo/scripts/setup-toolkit.cjs) is the script-backed English `setup toolkit` and `refresh toolkit` orchestrator, with shared workflow in `repo/scripts/setup-toolkit-core.cjs`. One semantic model drives the complete compact bank, terminal and piped answers, flags, plans, JSON, summaries, and execution. Final host/availability resolution derives contiguous section/question indexes, A-ZZ choice references, totals, and the quick index while stable semantic IDs and canonical choice values remain authoritative. Exact `all recommended` and changed-only indexed input bind canonical selections to the exact in-memory bank and fail closed on stale or inconsistent semantics. Active-to-managed delegation first performs a bounded no-probe protocol identity check, then immediately runs one managed child with captured stdout/stderr and private control/acknowledgement pipes. Non-TTY stdin is streamed asynchronously as exact bounded bytes, so child launch and bank visibility never wait for parent stdin EOF; fully explicit flags need no stdin read, while unresolved input is consumed only after bank acknowledgement. The child synchronously emits exactly one begin marker, canonical stdout bank, and complete marker, then reports bounded marker counts, question count, exact bank byte length, and SHA-256. The parent waits for that exact bounded stdout payload regardless of control/stdout pipe delivery order, validates length, digest, and markers, forwards it once, and only then acknowledges visibility. Only after acknowledgement may input waits, prompts, or writes continue. Missing, partial, misordered, duplicate, wrong-stream, length/digest mismatch, oversized input/output/metadata, timed-out, signalled, failed, recursive, or invalid-identity results fail closed with privacy-safe diagnostics and no retry or active-checkout fallback. Claude command selection is shared with plugin verification and launch: explicit current argument, `AI_AGENT_TOOLKIT_CLAUDE_CLI`, `CLAUDE_TOOLKIT_CLAUDE_CLI`, `CLAUDE_CLI_PATH`, persisted profile command, then bare default. The selected command fails closed rather than silently falling through. Claude automatic/manual direct choices require separate verified launch, resource-counter, installed-byte, trust, and active-hook capabilities. Launch capability uses a bounded empty-input, no-session-persistence invocation with the exact worker argument shape instead of relying on incomplete help text; unsupported syntax is distinguished from unrelated authentication/network/runtime failure and indeterminate results fail closed. Unsupported or malformed resource inspection omits direct capacity and resolves recommended behavior safely. A supported kept topology resolves before capacity reconciliation, so kept broader-native remains broader-native with root-only/not-applicable Toolkit capacity. Strict kept profiles are invalidated when any required capability is lost.

[repo/scripts/toolkit-agent-control.cjs](../../../repo/scripts/toolkit-agent-control.cjs) is the canonical Claude direct-agent launch and admission boundary. Before admission it re-reads native plugin state and requires current enabled, trusted, hook-active state plus proof-schema-2 hashes for the exact persisted cache, hooks configuration, controller, and executed `toolkit-claude-agent-hook.cjs`. Strict profiles preserve the original verified Claude command; launch precedence is explicit argument, current environment override, persisted command, then bare `claude`. Toolkit child environments refuse re-entry before probing or lock state. Admission locks and recovery markers publish private PID/time/token ownership; fresh or live-owned state remains protected, while stale dead, ownerless, or malformed state recovers after the bounded TTL. Reservation update/release validates the complete existing state before mutation and leaves malformed, future, contradictory, or partially invalid bytes untouched. The controller validates productive-root launch specifications, forces medium effort by default and non-fast child execution, denies both `Agent` and legacy `Task`, reads supported validated physical plus commit/pagefile counters, applies substantial reserves after subtracting aggregate live reservation costs, and returns only `start`, `queue`, or `refuse-root-only`. Exact UTF-8 prompt bytes are bounded before admission and serialized losslessly for stdin transport; oversized prompts create no admission or job state. [repo/scripts/claude-process-launch.cjs](../../../repo/scripts/claude-process-launch.cjs) provides the shared executable contract for setup, capability probes, and direct workers. Relative path-like values are rejected on every platform. POSIX bare `claude` remains unpinned for official launcher updates; Windows bare names resolve only from absolute PATH entries and the exact absolute candidate is carried into execution so repository cwd shims cannot shadow it. Explicit JavaScript, `.exe`, `.cmd`, and `.bat` paths retain their reviewed shell and escaping boundaries.

[repo/scripts/toolkit-claude-agent-hook.cjs](../../../repo/scripts/toolkit-claude-agent-hook.cjs) is the Claude `PreToolUse` boundary for native `Agent` and legacy `Task`. ### Setup Bank Approval Reference Amendment

The concise-answer in-memory binding described above applies directly only to a live TTY response entered after that process rendered the bank. Non-interactive and managed-continuation concise answers carry a displayed privacy-safe 80-bit bank reference derived from the full approval-relevant SHA-256 identity. The parser validates that reference before mapping any indexed question or choice, so host, order, current state, verification, availability, recommendation, choice, or after-applying drift fails closed. Live TTY setup presents one corrective concise-command stage before the one-at-a-time fallback; secondary path, retention, and advanced values are collected before approval or rejected with precise non-interactive instructions.
