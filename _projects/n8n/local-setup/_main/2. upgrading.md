# 2. Upgrading n8n

The primary local setup guide is [1. Local Setup](./1.%20local%20setup.md). This page is retained as an appendix/reference for focused upgrade details, not as a separate start path.

This guide covers the safest low-friction ways to update a self-hosted n8n instance.

It is written for five common setups:

* Hostinger Docker Manager / Docker Catalog installs.
* Hostinger VPS template installs.
* Standard Docker Compose installs on any VPS.
* Local Docker Compose installs like the setup in this repo.
* Local npm installs.

Important:

1. Back up first.
2. Read the release notes before big version jumps.
3. Do a quick smoke test after every upgrade.

---

## 1. Before You Update

Do these first:

1. Take a backup or snapshot.
2. Record the current n8n version.
3. Know where your `docker-compose.yml` or `.env` lives.
4. Expect a short restart window.

If your instance is business-critical, do the update during a quiet period.

---

## 2. Hostinger Docker Manager / Docker Catalog

This is the lowest-effort Hostinger route.

### Option A. Docker Manager UI

If your n8n instance was deployed through Hostinger Docker Manager:

1. Open Docker Manager.
2. Find the n8n project.
3. Use **Update**.

### Option B. SSH / Browser Terminal

1. For current Hostinger Docker Catalog deployments, the n8n project commonly lives under:

   ```bash
   cd /docker/n8n
   ```

2. Then run:

   ```bash
   docker compose pull
   docker compose down
   docker compose up -d
   ```

---

## 3. Hostinger VPS Template Path

If you used Hostinger's n8n VPS template, n8n comes preinstalled in Docker.

Hostinger's template guide currently points you to `/root/docker-compose.yml` for environment changes, so the compose project commonly lives under `/root`.

1. Typical update flow:

   ```bash
   cd /root
   docker compose pull
   docker compose down
   docker compose up -d
   ```

   A) If your files live elsewhere, update from the folder that contains your `docker-compose.yml`.

---

## 4. Standard Docker Compose Upgrade On Any VPS

1. If you are running a normal Docker Compose based install on any provider:

   ```bash
   cd /path/to/your/compose/project
   docker compose pull
   docker compose down
   docker compose up -d
   ```

   A) That is also the flow used in n8n's official Docker Compose based VPS guides.

---

## 5. Local Docker Compose Upgrade ( This Repo Style )

1. If you followed the local setup from this repo, update from the local stack folder that contains [docker-compose.yml](./templates/local-stack/docker-compose.yml) and the local `.env`.

2. Run this in PowerShell:

   ```powershell
   cd "$env:USERPROFILE\Desktop\n8n-local"
   docker compose pull
   docker compose up -d --force-recreate
   ```

3. Notes:

   A) The `n8n_data` and `postgres_data` Docker volumes keep local runtime data during the recreate.
   B) `WEBHOOK_URL`, `N8N_HOST`, `N8N_PROTOCOL`, and `N8N_PROXY_HOPS` stay in `.env`.
   C) ngrok is the only supported local tunnel path in this guide, and it runs as a Compose service.
   D) Do not commit `.env`, credentials, runtime payloads, `.n8n-local/`, `.tmp/`, or live n8n imports/exports.

---

## 6. If You Installed n8n With npm

1. If you are running a local or direct npm install, the official n8n update command is:

   ```bash
   npm update -g n8n
   ```

2. If you need to go to a specific version:

   ```bash
   npm install -g n8n@<version>
   ```

---

## 7. Verify The Upgrade

### Check containers

1. ```bash
   docker compose ps
   ```

### Check version from inside the container

1. For Compose setups, first find the n8n container name, then run:

   ```bash
   docker exec -it <container_name> n8n -v
   ```

### Check the app itself

After the stack comes back:

1. Open the n8n editor.
2. Trigger a simple workflow.
3. Confirm webhooks still load.
4. Confirm community nodes still appear if you use them.

---

## 8. Rollback Plan

If you want easier rollbacks, pin a specific n8n image tag instead of always tracking `stable` or `latest`.

That makes it much easier to revert after a bad update. If something breaks, switch the image tag back to the last known-good version and restart the stack.

If the upgrade involved a database migration and you need to revert an npm install, check the n8n update docs and release notes before rolling back.

---

## 9. Post-Upgrade Smoke Test

Run this quick checklist:

1. Login works.
2. Credentials still decrypt correctly.
3. Webhook URLs are still correct.
4. Scheduled workflows still run.
5. Community nodes still load.
6. Queue workers reconnect properly if you use queue mode.

If you use a reverse proxy, also confirm:

7. `WEBHOOK_URL` is still correct.
8. `N8N_PROXY_HOPS=1` is still set.

---

## 10. When Not To Update Immediately

If your instance runs revenue-critical or client-facing automations, do not update blindly the moment a new release lands.

Wait long enough to read the release notes, then update with a backup or snapshot ready.
